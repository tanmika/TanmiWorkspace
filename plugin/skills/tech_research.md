# 技术调研

**核心思维**: 权衡 (Trade-off)
**典型动作**: 方案对比、查阅最佳实践

## 执行步骤 (SOP)

1. **识别选项**
   - 基于内部知识库列出候选方案（至少2个）
   - 收集每个方案的核心特性
   - 记录官网/仓库链接

2. **约束匹配**
   - 检查项目现有技术栈
   - 排除不兼容的方案
   - 验证环境要求（Node版本、依赖等）

3. **多维度对比**
   - **实现复杂度**: 代码量、配置复杂度
   - **性能表现**: 吞吐量、延迟、资源占用
   - **维护成本**: 学习曲线、文档质量、社区支持

4. **推荐结论**
   - 给出明确的建议方案
   - 说明选择理由（至少3点）
   - 标注前置条件和风险

## 检查清单

### 方案完整性
- [ ] 至少识别2个候选方案
- [ ] 每个方案有官网/仓库链接
- [ ] 记录了核心特性和适用场景

### 约束验证
- [ ] 检查了与现有技术栈的兼容性
- [ ] 验证了环境要求（语言版本、依赖）
- [ ] 排除了明确不符合的方案

### 对比维度
- [ ] **实现复杂度**: 评估了开发工作量
- [ ] **性能表现**: 对比了性能指标
- [ ] **维护成本**: 评估了长期维护负担

### 结论质量
- [ ] 给出了明确的推荐方案
- [ ] 说明了选择理由（至少3点）
- [ ] 标注了前置条件
- [ ] 识别了主要风险并提供缓解措施

## 验证规则

### 完整性检查
- 至少 2 个候选方案
- 每个方案包含：官网/仓库、核心特性、适用场景
- 对比表包含 3 个核心维度（实现复杂度、性能表现、维护成本）
- Pros/Cons 对比完整

### 客观性检查
- 对比有具体依据（数据、官方文档、实际案例）
- 优缺点对比客观公正
- 避免主观臆断和模糊描述

### 可操作性检查
- 推荐结论明确
- 理由至少3点且与需求强相关
- 前置条件可验证
- 风险有具体缓解措施

### 完成标准
- 所有对比维度已评估
- Pros/Cons 对比表完整
- 给出明确推荐结论
- 标注了前置条件和风险缓解方案
