# 技术调研

## 执行步骤

1. **明确调研目标**
   - 确定要解决的技术问题
   - 列出关键技术需求
   - 定义评估标准

2. **收集候选方案**
   - 搜索现有技术方案（至少2个）
   - 查阅官方文档、社区讨论
   - 收集实际案例

3. **多维度评估**
   - 按调研维度逐项评分
   - 记录优缺点
   - 识别风险点

4. **输出对比报告**
   - 填写技术对比表
   - 给出选型建议
   - 标注不确定项

## 检查清单

### 可行性评估
- [ ] **技术成熟度**：是否已有生产环境案例？
- [ ] **生态完整性**：文档、工具、插件是否齐全？
- [ ] **兼容性**：是否与现有技术栈兼容？
- [ ] **性能表现**：是否满足性能要求？

### 社区与支持
- [ ] **社区活跃度**：GitHub Star/Issue 数量？最后更新时间？
- [ ] **维护状态**：是否有持续维护？团队/公司背书？
- [ ] **问题解决能力**：遇到问题能否快速找到答案？

### 成本评估
- [ ] **学习成本**：团队需要多久上手？
- [ ] **迁移成本**：从现有方案迁移难度？
- [ ] **维护成本**：长期维护工作量？
- [ ] **License 成本**：是否有许可证限制/费用？

### 风险识别
- [ ] **技术风险**：是否有已知Bug/限制？
- [ ] **依赖风险**：依赖链是否稳定？
- [ ] **替代方案**：如果失败，有Plan B吗？

## 调研维度

### 1. 可行性（Feasibility）
- **技术原理**：核心实现机制
- **适用场景**：最佳使用场景与限制
- **性能指标**：吞吐量、延迟、资源占用

### 2. 成熟度（Maturity）
- **版本状态**：稳定版本、发布频率
- **生产案例**：知名公司/项目使用情况
- **Bug 情况**：已知问题、修复速度

### 3. 社区活跃度（Community）
- **Star/Fork 数**：GitHub 指标
- **提交频率**：最近3个月提交情况
- **Issue 响应**：问题响应速度、解决率

### 4. 学习成本（Learning Curve）
- **文档质量**：是否有完整文档、教程
- **上手难度**：HelloWorld 到实际应用的时间
- **团队熟悉度**：团队是否有相关经验

## 验证规则

### 完整性检查
- 至少 2 个候选方案
- 每个方案包含：官网、核心特性、适用场景
- 对比表包含 4 个维度评分

### 客观性检查
- 评分有具体依据（数据、案例）
- 优缺点对比客观公正
- 避免主观臆断

### 可操作性检查
- 选型建议明确
- 前置条件可验证
- 风险有缓解措施

### 完成标准
- 所有调研维度已评估
- 技术对比表完整
- 给出明确选型建议
- 用户对调研结果已确认
